# Andy Code Cat ‚Äî BaaS Services Layer: Specification

> **Status:** specification approved ‚Äî deferred after R3/R5 (mature Layer 1)
> **Architectural dependencies:** double sandbox middleware (‚úÖ implemented), Redis rate‚Äëlimiting (‚úÖ), nginx wildcard (R5)
> **Guiding principle:** static pages generated by Layer 1/2 acquire dynamic behaviours
> through calls to the Andy Code Cat BaaS ‚Äî without the user writing any server-side code
> and without Andy Code Cat managing per-tenant runtime infrastructure.

---

## 1. Problem and solution

### 1.1 Gap between SOL‚Äë1 and SOL‚Äë3

Layer 1 produces high-quality static pages (SOL‚Äë1). The next step in the original vision
(SOL‚Äë3, full web app) requires per-tenant runtime infrastructure ‚Äî effectively a mini-PaaS
built from scratch, expensive in ops and attack surface.

**The BaaS layer solves 90% of SOL‚Äë3 use cases without any per-tenant runtime infrastructure:**

| What the user wants | SOL‚Äë3 solution (discarded) | BaaS solution |
|---|---|---|
| Contact / sign-up form | Per-tenant Express endpoint | `PF.forms.submit()` ‚Üí shared API |
| Product checkout | Per-tenant Node.js server with Stripe | `PF.payment.checkout()` ‚Üí shared API |
| Telegram order notification | Per-tenant Telegram bot | `PF.telegram.notify()` ‚Üí shared relay |
| CRM / email automations | Per-tenant custom logic | `PF.webhook.trigger()` ‚Üí n8n/Make relay |
| Editable product catalogue | Per-tenant database | BaaS API: `ProductService` |

### 1.2 The remaining 10%

Use cases not covered by the BaaS (real-time analytics dashboards, site-user authentication,
real-time chat, social features) fall outside the target use cases of Andy Code Cat (landing pages,
portfolios, basic e-commerce, professional sites). Users who need those are better served by
dedicated tools.

### 2.2 Data entities

#### ProjectServiceConfig (embedded in Project)

```typescript
interface ProjectServiceConfig {
  // Public key embeddable in HTML ‚Äî not a secret
  publicKey: string;           // UUID v4, generated on project creation
  // Authorised CORS origins ‚Äî updated automatically on publish
  allowedOrigins: string[];    // e.g. ["https://abc123.Andy Code Cat.io", "https://www.example.com"]
  enabledServices: ServiceType[];  // ["forms", "payments", "telegram", "webhook", "catalog"]
  // Per-service configuration
  services: {
    forms?: FormsServiceConfig;
    payments?: PaymentServiceConfig;
    telegram?: TelegramServiceConfig;
    webhook?: WebhookServiceConfig;
    catalog?: CatalogServiceConfig;
  };
}

type ServiceType = 'forms' | 'payments' | 'telegram' | 'webhook' | 'catalog';
```

## 3. Business model for services

### 3.1 Two operating modes for each service

Andy Code Cat supports two modes for each integrable service, selectable per project:

| Mode | Who owns the external account | Who pays external fees | Andy Code Cat earns | Setup required |
|---|---|---|---|---|
| **BYOK** ‚Äî Bring Your Own Keys | The user/owner | The user | Monthly plan fee | Insert keys in the dashboard |
| **Managed** ‚Äî Andy Code Cat key | Andy Code Cat | Andy Code Cat (passed on to user) | Transaction fee % | Simple onboarding only |

### 3.2 Per-service analysis

#### Payments (Stripe)

**BYOK:**

- The user has their own Stripe account (requires identity verification)
- Money goes directly to the user's Stripe account
- Andy Code Cat never touches the funds
- Suitable for: professionals, businesses, anyone doing significant volume
- Andy Code Cat fee: included in the monthly plan

**Managed (Stripe Connect):**

- Andy Code Cat has a Stripe Platform account
- Each Andy Code Cat owner who activates payments goes through a Stripe Express onboarding (5 min, simplified KYC)
- Stripe creates a Connected Account linked to the Andy Code Cat Platform account
- Payments go through the Platform account, with `transfer_data.destination` to the owner's Connected Account
- The owner receives funds to their bank account; Andy Code Cat retains the application fee
- **Suitable for:** hobbyists, small sellers, anyone who doesn't want to set up a standalone Stripe account

**Managed fee structure (example):**

```
‚Ç¨100 transaction:
  Stripe fee (EU cards): ~1.5% + ‚Ç¨0.25 = ‚Ç¨1.75
  Andy Code Cat platform fee: 1.5% = ‚Ç¨1.50
  Owner receives: ‚Ç¨96.75
  Total cost to the owner: 3.25% ‚Äî competitive with PayPal (3.49%) and Shopify Payments (2%)
```

**IMPORTANT tax note:**
Andy Code Cat as a platform does not issue invoices for the owner's sales ‚Äî Stripe handles
tax reporting directly. Andy Code Cat issues only its own service invoice for the platform fee.
The owner is responsible for their own accounting.

For small hobbyists and micro-businesses below VAT registration thresholds, the Managed Stripe
model is the path of least resistance: no business account needed, identity verified directly
through Stripe Express, personal bank account.

#### Telegram

**BYOK:**

- The user creates their own bot via @BotFather (1 minute, free)
- Inserts the token in the dashboard
- The bot is owned by the user; Andy Code Cat is only the relay
- Recommended: the user has more control, no dependency on the Andy Code Cat bot

**Managed:**

- Andy Code Cat has a shared relay bot
- The owner sends `/start` to the Andy Code Cat bot, obtaining their `chatId`
- All notifications go through the relay bot (e.g. `@Andy Code CatBot`)
- Downside: the bot can be pinged by anyone who knows the chatId (mitigated with OTP token for subscriptions)
- Recommended only as a simplified fallback for users unfamiliar with Telegram bots

**Recommendation:** BYOK Telegram is so simple that Managed adds no real value.
Offer Managed as the "simple default" with documented migration to BYOK.

#### n8n / Make / Webhook

Only BYOK makes sense: each user has their own n8n, Make, Zapier, or custom webhook instance.
Andy Code Cat only POSTs to the configured URL. No advantage to Managed.

#### Forms

Fully managed by Andy Code Cat (no external account required).
Submissions are stored in MongoDB in the project namespace.
Notification email via Andy Code Cat SMTP/Resend (included in the base plan).

## 4. Security ‚Äî Configuration Guidelines

### 4.1 BaaS layer security principles

The BaaS is exposed to requests from anonymous visitors on the web ‚Äî the attack surface
is different from that of Andy Code Cat's authenticated APIs.

**Fundamental rule:** no secret must ever leave the server. The client receives only
`projectPublicKey` (not a secret) and sanitised response data.

### 4.2 Configuration guidelines for user/project profile

#### Level 1 ‚Äî Identity and access protection (MANDATORY)

| Control | Implementation | Notes |
|---|---|---|
| **CORS locked** | `allowedOrigins` verified for every public request | Updated automatically on publish. The owner can add custom domains. Wildcard `*` never allowed. |
| **Rate limiting per projectKey** | Redis sliding window: 60 req/min per `projectKey`, 10 req/min per `projectKey+ipHash` | Flood and submission-scraping prevention |
| **HTTPS only** | All SDK calls over HTTPS | The SDK rejects non-SSL calls in production |
| **IP hashing** | Salted SHA-256 of the IP for rate limiting | The real IP is never persisted in the DB |

#### Level 2 ‚Äî Input validation (MANDATORY)

| Attack | Defence |
|---|---|
| **Form injection (stored XSS)** | All form data is sanitised with a DOMPurify-equivalent on the server before saving. No HTML field accepted in forms. Only strings, numbers, booleans. |
| **Oversized payload** | 16 KB limit per submission. 50 items limit per checkout. 1000 char limit per form field. |
| **Field injection** | Only fields present in `FormsServiceConfig.forms[].allowedFields` are saved. Extra fields are silently ignored. |
| **SQL/NoSQL injection** | MongoDB: parameters passed as typed values, never interpolated into query strings. Mongoose schema validation before every write. |
| **Command injection** | Not applicable: the BaaS router does not execute user code. |

#### Level 3 ‚Äî Secrets protection (MANDATORY for BYOK)

| Rule | Detail |
|---|---|
| **Stripe secret key never in MongoDB** | BYOK secret keys are encrypted with AES-256-GCM using an envelope key (env var `SECRET_VAULT_KEY`). Only the reference ID is saved in the DB. Plaintext is decrypted in memory only at point of use. |
| **Telegram bot token** | Same mechanism: envelope encryption, reference stored in DB. |
| **No secret logging** | The logging middleware explicitly excludes fields matching `*Key`, `*Token`, `*Secret` from logs. |
| **Secret rotation** | The owner can rotate their BYOK keys from the dashboard without downtime. The old reference is invalidated immediately. |
| **Principle of least privilege** | The service worker using the Stripe secret key has read access only to that specific record ‚Äî not to the entire ProjectServiceConfig collection. |

#### Level 4 ‚Äî Sandbox and isolation (MANDATORY)

| Rule | Detail |
|---|---|
| **Double sandbox on every write** | Even requests via `publicKey` are tracked with `projectId + ownerId`. No submission or order can be created without this pair. |
| **Cross-project isolation** | `publicKey` resolves to exactly one project. It is impossible to access another project's data via a different `publicKey`. |
| **Form submissions visible to owner only** | Submission read APIs (`GET /v1/projects/:id/services/forms/:formId/submissions`) are protected by `authMiddleware` + `projectSandboxMiddleware` ‚Äî identical to other project APIs. |
| **Stripe webhook signature** | Every Stripe callback is verified with `stripe.webhooks.constructEvent()` + webhook secret. Unsigned requests are rejected with 400. |

#### Level 5 ‚Äî Audit and monitoring

| Control | Implementation |
|---|---|
| **Execution logging** | Every BaaS layer call generates an event in `execution_logs` (TTL-90d collection ‚Äî already implemented) with `domain: "baas"`, `eventType`, `projectId`, `ipHash` |
| **Anomaly detection (future)** | Abnormal submission rate for a project ‚Üí owner email alert |
| **Abuse reporting** | Endpoint `POST /v1/public/report-abuse` for reporting misuse of an Andy Code Cat site |

## 5. API ‚Äî Endpoints

### 5.1 Public BaaS API (no JWT ‚Äî calls from anonymous visitors)

```
POST /v1/public/svc/:projectKey/forms/:formId/submit
POST /v1/public/svc/:projectKey/payments/checkout
POST /v1/public/svc/:projectKey/telegram/notify
POST /v1/public/svc/:projectKey/webhook/trigger/:workflowId
GET  /v1/public/svc/:projectKey/catalog/list
GET  /v1/public/svc/:projectKey/catalog/item/:itemId
```

**Request/Response per forms:**

```typescript
// POST /v1/public/svc/:projectKey/forms/contact/submit
// Body (Content-Type: application/json, max 16KB):
{
  "name": "John Smith",
  "email": "john@example.com",
  "message": "Hello, I would like some information"
}

// Response 201:
{ "success": true, "submissionId": "uuid" }

// Response 429 (rate limit):
{ "error": "too_many_requests", "retryAfter": 3600 }

// Response 403 (unauthorised origin):
{ "error": "origin_not_allowed" }
```

**Request/Response per payments:**

```typescript
// POST /v1/public/svc/:projectKey/payments/checkout
{
  "items": [
    { "name": "Product A", "quantity": 2, "unitAmount": 2500 }  // in cents
  ],
  "customerEmail": "john@example.com",  // optional
  "metadata": { "sourceForm": "product-page" }  // max 5 kv pairs
}

// Response 201:
{
  "checkoutUrl": "https://checkout.stripe.com/pay/cs_...",
  "orderId": "uuid"
}
```

### 5.3 SDK Client ‚Äî Public interface

```typescript
// Initialisation (automatic when the script is loaded with ?pk=...)
window.PF.ready(() => {
  // SDK ready
});

// Form submit
const result = await PF.forms.submit('contact', {
  name: formData.get('name'),
  email: formData.get('email'),
  message: formData.get('message')
});
// result: { success: boolean, submissionId?: string, error?: string }

// Payment checkout (redirect)

  { name: 'Product name', quantity: 1, unitAmount: 4990 }
]);
// ‚Üí redirects to Stripe Checkout (no return ‚Äî it's a redirect)

// Telegram notify (only if enabled for the project)

// Webhook trigger
await PF.webhook.trigger('send-to-crm', { email, name });

// Catalog (read-only from client)
const items = await PF.catalog.list({ page: 1, limit: 20 });
```

## 6. Integration with the prompt system

### 6.1 Layer A extension (SDK injection)

When the project has BaaS services enabled, `buildBaseConstraintsLayer()` receives
an additional `baasContextBlock`:

```
[Andy Code Cat SERVICES SDK ‚Äî injected only if project.services.enabledServices.length > 0]

This project uses the Andy Code Cat BaaS SDK for dynamic behaviours.
Script already included: <script src="https://api.Andy Code Cat.io/sdk/v1.js?pk={projectKey}"></script>

Use ONLY these APIs for dynamic functionality ‚Äî do not generate custom fetch() calls, do not use
third-party services directly (Stripe.js, emailjs, formspree, etc.):

  PF.forms.submit(formId, data)      ‚Üí submit a form (formId: "{formIds}")
  PF.payment.checkout(items)         ‚Üí start Stripe Checkout (if enabled)
  PF.telegram.notify(message)        ‚Üí notify Telegram owner (if enabled)
  PF.webhook.trigger(workflowId, data) ‚Üí trigger automation (workflowId: "{workflowIds}")

Every HTML form must call PF.forms.submit() in the submit handler.
Do not use action="" on forms. Handle the submit with JavaScript.
Show visual feedback (loading, success, error) after every SDK call.
```

## 9. Architectural decision: BYOK first, Managed later

The recommended path is:

1. **BS1 (Forms)** ‚Äî zero infra cost, 70% of sites need it, zero legal risk
2. **BS2 (Telegram BYOK)** ‚Äî only requires the user to create a Telegram bot (1 min)
3. **BS3 (Payments BYOK)** ‚Äî Stripe BYOK is usable by anyone with a registered business; requires no complex infra from Andy Code Cat
4. **BS4 (Payments Managed)** ‚Äî requires: opening a Stripe Platform account, agreement with Stripe on marketplace terms, KYC compliance for Connected users ‚Üí This is the step with the most legal and compliance work, not technical. It must be planned separately alongside the commercial/legal side.

> **BS4 is not just a technical milestone** ‚Äî it requires Andy Code Cat to be registered as a Payment Facilitator or Marketplace under Stripe's terms. This has fiscal and liability implications. Validate with an advisor before implementing.
