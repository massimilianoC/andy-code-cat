## 1. Executive Decision

The platform must stop treating image URLs generated by the LLM as the primary media contract.

The new primary contract is:

```text
LLM generates semantic media requests, not provider URLs.
Backend resolves media requests deterministically.
Every resolved media file becomes a ProjectAsset.
Artifacts and snapshots store stable internal media references.
```

The LLM remains responsible for:

- page/content semantics;
- visual intent;
- media role and context;
- suggested keywords and image prompts;
- placement keys inside the artifact.

The backend becomes responsible for:

- provider choice;
- fallback policy;
- API-key resolution;
- stock-image retrieval;
- future image generation provider selection;
- project/user library lookup;
- file download/storage;
- `ProjectAsset` creation;
- placeholder replacement;
- trace, audit, notifications, and version linkage.

This is a deliberate beta-stage refactor. Backward compatibility is required for already persisted/public artifacts, but not as a reason to keep the old URL-driven system as the main path.

---

## 2. Non-Negotiable Product Rules

1. LLM output must not depend on hardcoded stock provider URLs.
2. New generated artifacts must use deterministic media placeholders.
3. New generated artifacts must include a typed media manifest.
4. Provider selection must come from superadmin policy, not from the LLM and not from the frontend.
5. Every resolved image used in a preview snapshot must be persisted as a `ProjectAsset`.
6. Preview snapshots must reference stable internal media URLs, never random provider URLs.
7. Publication must publish the selected/current artifact version exactly as resolved.
8. Edit-mode image replacement must reuse the existing media control popup/accordion flow.
9. Fallback to LoremFlickr/Picsum is an operational degradation, not normal success.
10. If fallback is used, user and superadmin notification must be created.
11. If fallback is disabled and the selected provider fails, generation must fail visibly.
12. Frontend must not implement provider, fallback, or asset-resolution logic.

## 3. Target Flow

### 3.1 Artifact Generation Flow

```text
User prompt
  -> LLM orchestration
  -> structured artifact output:
       artifacts.html/css/js
       mediaManifest.requests[]
  -> ArtifactMediaOrchestrator
       validate manifest
       validate placeholders
       resolve each media request
       persist ProjectAsset
       write MediaResolutionTrace
       replace placeholders in HTML/CSS
       emit notifications on degraded/error paths
  -> PreviewSnapshot created/activated with stable media URLs
```

#### 3.1.1 As-built progress feedback (implemented 2026-05-29)

The deterministic resolution runs **after** the LLM stream completes and **before** the
`done` SSE event, emitting step-by-step `media_progress` SSE events so the chat UI shows
live feedback. Resolution is parallel and non-exclusive (one failure does not abort the rest).

```text
LLM stream (thinking/answer) -> stream ends
  -> ResolveArtifactMedia.execute({ onProgress })
       emit media_progress { phase: "start", total }
       for each request (in parallel):
         emit { phase: "resolving", mediaKey }
         fetch stock from configured provider (policy order)
         persist ProjectAsset
         emit { phase: "resolved", index, total, provider, fallbackUsed }
                or { phase: "failed", index, total }
       emit { phase: "replacing" }      // swap asset://media/<key> -> /p/media/:id in HTML+CSS
       emit { phase: "done", resolvedCount, total }
  -> SSE done { result with mediaResolution metadata }
```

UI mapping: `apps/web` `streamLlmChatPreview` forwards `media_progress` to the workspace,
which renders a single live-updating system notification (start → resolving → resolved N/total
→ replacing → done). Fallback/failure also emit persistent `SystemNotification` records.

Note: this is **in-request progress feedback over SSE**, not a detached background job. A fully
detached background-job model (chat returns placeholders immediately, worker patches the snapshot
later) remains future work tracked in `ZERO_EFFORT_MEDIA_ASYNC_EVOLUTION_SPEC.md`. The non-stream
`POST /chat-preview` path resolves synchronously without progress events (single JSON response).

### 3.2 Edit-Mode Media Flow

```text
User selects image/background in Edit mode
  -> existing media inspector popup opens
  -> user chooses:
       regenerate stock
       generate image
       choose project asset
       upload/select library asset
  -> backend ArtifactMediaOrchestrator resolves/replaces media
  -> frontend applies returned internal asset URL
  -> saveMediaVersion creates a new PreviewSnapshot
```

### 3.3 Publication Flow

```text
Selected/active PreviewSnapshot
  -> publish/export flow reads already-resolved artifact
  -> no provider call during publication
  -> published site contains stable internal/copyable media references
```

Publication must not perform stock-image resolution. Publication must be deterministic from the selected snapshot.

## 4. Structured LLM Output Contract

### 4.1 Placeholder Markup

HTML generated by the LLM must use media placeholders:

```html
<img
  src="asset://media/hero-main"
  data-media-key="hero-main"
  alt="Creative team working in a bright design studio"
/>
```

For CSS backgrounds:

```html
<section data-media-key="hero-background" class="hero"></section>
```

```css
.hero {
  background-image: url("asset://media/hero-background");
}
```

Allowed placeholder scheme:

```text
asset://media/<mediaKey>
```

Rules:

- `mediaKey` must be unique inside the artifact.
- `mediaKey` must be lowercase kebab-case.
- Every `asset://media/<mediaKey>` must have one manifest request.
- Every manifest request should be referenced by HTML or CSS.
- `alt` must be meaningful for foreground images.
- LLM must not emit `loremflickr.com`, `picsum.photos`, `pexels.com`, `pixabay.com`, `unsplash.com`, or provider-specific stock URLs in new outputs.

### 4.2 Media Manifest

Add to structured LLM responses:

```ts
export interface ArtifactMediaManifest {
  version: "media-manifest-v1";
  requests: ArtifactMediaRequest[];
}
```

```ts
export type MediaSourceStrategy =
  | "auto"
  | "stock"
  | "image_generation"
  | "project_asset"
  | "user_library";

export type ArtifactMediaKind =
  | "image"
  | "background"
  | "logo"
  | "icon"
  | "avatar"
  | "decorative";

  | "hero"
  | "section"
  | "card"
  | "gallery"
  | "testimonial"
  | "avatar"
  | "background"
  | "logo"
  | "icon"
  | "decorative";

export interface ArtifactMediaRequest {
  key: string;
  kind: ArtifactMediaKind;
  role: ArtifactMediaRole;
  sourceStrategy: MediaSourceStrategy;

  /**
   * Short semantic search text for stock providers and project-library retrieval.
   * This is not provider syntax.
   */
  semanticQuery: string;

  /**
   * Richer creative prompt for image-generation providers.
   * Optional for stock-only images.
   */
  generationPrompt?: string;

  alt: string;
  width?: number;
  height?: number;
  aspectRatio?: number;
  priority: number;

  constraints?: {
    noText?: boolean;
    noLogo?: boolean;
    safeCrop?: boolean;

    avoid?: string[];

    pageSection?: string;
    nearbyHeading?: string;
    nearbyText?: string;
    brandTone?: string;
  };
}
```

### 4.3 Structured Response Shape

Extend `LlmStructuredResponse` or equivalent:

```ts
interface LlmStructuredArtifactResponse {
  chat: {
    summary: string;
    bullets: string[];
    nextActions: string[];
  };
  artifacts: {
    html: string;
    css: string;
    js: string;
  };
  mediaManifest?: ArtifactMediaManifest;
}
```

During migration, `mediaManifest` can be optional for legacy outputs. For new primary generation prompts, it must be required by prompt instruction and validated.

## 5. Backend Orchestrator

### 5.1 New Central Use Case

Create:

```text
apps/api/src/application/media/ResolveArtifactMedia.ts
```

Suggested signature:

```ts
interface ResolveArtifactMediaInput {
  projectId: string;
  userId: string;
  artifacts: {
    html: string;
    css: string;
    js: string;
  };
  mediaManifest?: ArtifactMediaManifest;
  sourceContext: {
    route:
      | "chat-preview"
      | "chat-preview-stream"
      | "focus-patch"
      | "edit-regenerate"
      | "manual-asset-apply";
    conversationId?: string;
    parentSnapshotId?: string;
    selectedElementSelector?: string;
  };
  mode: "initial_generation" | "focused_edit" | "manual_regeneration";
}

interface ResolveArtifactMediaResult {
  artifacts: {
    html: string;
    css: string;
    js: string;
  };
  resolvedAssets: ProjectAsset[];
  traces: MediaResolutionTrace[];
  warnings: MediaResolutionWarning[];
}
```

Responsibilities:

1. Validate manifest schema.
2. Extract `asset://media/<key>` placeholders from HTML and CSS.
3. Match placeholders to manifest requests.
4. Reject missing manifest requests in strict mode.
5. Resolve every request through `MediaResolverRegistry`.
6. Persist resolved files via existing storage and `ProjectAsset`.
7. Replace placeholders with stable internal media URLs.
8. Write `MediaResolutionTrace`.
9. Emit `ExecutionLogger`.
10. Emit persistent `SystemNotification` for fallback/error paths.

### 5.2 Resolver Registry

Create:

```text
apps/api/src/application/media/MediaResolverRegistry.ts
```

Resolvers:

```text
StockImageResolver
ImageGenerationResolver
ProjectAssetResolver
UserLibraryResolver
```

Interface:

```ts
interface MediaResolver {
  strategy: MediaSourceStrategy;
  canResolve(request: ArtifactMediaRequest, context: MediaResolutionContext): boolean;
  resolve(request: ArtifactMediaRequest, context: MediaResolutionContext): Promise<ResolvedMedia>;
}

interface ResolvedMedia {
  buffer?: Buffer;
  mimeType?: string;
  sourceUrl?: string;
  providerId: string;
  providerKind: "stock" | "image_generation" | "project_asset" | "user_library";
  attribution?: string;
  existingAssetId?: string;
  width?: number;
  height?: number;
  attemptedProviders: MediaProviderAttempt[];
  fallbackUsed: boolean;
}
```

Resolution rules:

- `auto`: backend chooses strategy based on policy and request role.
- `stock`: use stock provider policy.
- `image_generation`: use image-generation provider policy.
- `project_asset`: search existing project assets.
- `user_library`: search user library.

If no resolver succeeds:

- if operational fallback enabled, fallback resolver may produce a low-grade image and notify;
- otherwise fail the artifact generation or manual regeneration request.

## 6. Provider Policy

### 6.1 Single Source Of Truth

Policy must be stored server-side and managed by superadmin.

Recommended initial location:

```text
PlatformConfig.mediaProviderPolicy
```

Reason: avoids a new configuration island while the platform is still beta.

Future extraction to a dedicated collection is acceptable only if policy becomes complex enough to need per-provider history.

### 6.2 Policy Shape

```ts
export interface MediaProviderPolicy {

  stock: {
    defaultProvider: "pexels" | "pixabay" | "unsplash" | "loremflickr";

    allowNoKeyProviders: boolean;

  imageGeneration: {
    enabled: boolean;
    defaultProvider?: "siliconflow";
    fallbackMode: "disabled" | "notify";

    persistEveryResolvedMedia: true;
    strictManifest: boolean;
    strictPersistence: boolean;

    maxDownloadBytes: number;
  };

  notifications: {
    notifyUserOnFallback: boolean;
    notifyAdminOnFallback: boolean;
    notifyUserOnFailure: boolean;
    notifyAdminOnFailure: boolean;
  };
}
```

## 7. Persistence And Versioning Model

### 7.1 ProjectAsset Remains The File Source Of Truth

Every resolved media file must create or reuse a `ProjectAsset`.

Required metadata:

```ts
generationMetadata: {
  provider: string;
  model?: string;
  requestedAt: Date;
  completedAt?: Date;
  finishReason:
    | "media-manifest-stock-resolved"
    | "media-manifest-image-generated"
    | "media-manifest-project-asset-selected"
    | "media-regenerated";
  sourceUrl?: string;
  outputMimeType?: string;
  width?: number;
  height?: number;
  providerResponse: {
    mediaKey: string;
    mediaRequest: ArtifactMediaRequest;
    attribution?: string;
    fallbackUsed: boolean;
    attemptedProviders: MediaProviderAttempt[];
    sourceContext: Record<string, unknown>;
  };
}
```

### 7.2 MediaResolutionTrace Collection

Add:

```text
media_resolution_traces
```

Entity:


```text

{ status: 1, createdAt: -1 }
```

### 7.3 PreviewSnapshot Metadata

Extend snapshot metadata without changing artifact shape:

```ts
metadata: {
  finishReason: string;
  mediaResolution?: {
    version: "media-resolution-v1";
    traceIds: string[];
    assetIds: string[];
    mediaKeys: string[];
    degraded: boolean;
  };
}
```

Guardrail:

```text
PreviewSnapshot artifacts must be saved only after media placeholders are resolved or explicitly marked failed.
```

Do not save snapshots containing unresolved `asset://media/` placeholders unless the snapshot is an internal failed/debug record and not activated.

### 7.4 Publication Guardrail

Before publish/export:

1. Read the selected/active snapshot.
2. Scan HTML/CSS for unresolved `asset://media/`.
3. If unresolved placeholder exists:
   - fail publish/export with actionable error;
   - do not auto-resolve during publish;
   - notify user/admin.

Reason: publication must be deterministic and must not change the selected version.

| Event | User | Superadmin | Severity |
| --- | ---: | ---: | --- |
| Selected stock provider fails and fallback is used | yes | yes | warning |
| Selected stock provider fails and fallback disabled | yes | yes | error |

```text
Errore nel recupero immagini stock da Pexels. Fallback su LoremFlickr applicato.
```


## 9. Prompting Guidelines

### 9.1 System Prompt Rules

Add to artifact-generation prompt:

```text
## MEDIA CONTRACT

Do not embed external image provider URLs.
Do not use loremflickr.com, picsum.photos, pexels.com, pixabay.com, or unsplash.com.
For each required image, insert a deterministic placeholder:
  asset://media/<kebab-case-key>
Also add one mediaManifest request with the same key.
The backend will resolve the media request through configured providers.
```

### 9.2 Media Request Quality Rules

The LLM should:

- write semanticQuery as provider-agnostic search text;
- write generationPrompt only if image_generation may be useful;
- include role and context;
- include width/height or aspectRatio when layout implies it;
- prefer one strong image request over many vague decorative requests;
- mark decorative images as `decorative`;
- avoid text-in-image unless explicitly required.

### 9.3 Max Request Rules

Default:

```text
maxMediaRequestsPerArtifact = 20
```

Prompt should prefer:

- hero image;
- section backgrounds;
- gallery/card images;
- avatars/logos only if requested.

It should not request dozens of filler images.

## 10. Frontend Integration Rules

### 10.1 Workspace/Edit Mode

All image controls must remain inside the existing media inspector flow:

- current selected-image popup;
- existing accordion pattern;
- existing gallery/upload controls;
- existing apply/save media version flow.

Allowed UI extensions:

- accordion section: `Source`;
- accordion section: `Regenerate`;
- provider status badge;
- fallback warning indicator;
- buttons with lucide icons.

Not allowed:

- separate media-management modal unrelated to current inspector;
- frontend-side provider choice logic;
- frontend-side fallback logic;
- frontend-side direct provider API calls.

### 10.2 Existing Actions To Preserve

Must keep working:

- apply existing project asset;
- upload media through existing upload endpoint;
- choose from user media library;
- AI generate image through existing generation flow;
- apply current styles;
- save new preview version after media replacement.

### 10.3 New Edit Actions

Target endpoint:

```http
POST /v1/projects/:projectId/media/:mediaKey/regenerate
```

Fallback for selected DOM without manifest:

```http
POST /v1/projects/:projectId/media/regenerate-selected
```

The backend receives:

```ts
{
  selectedElement: LlmFocusContext["selectedElement"];
  currentSnapshotId?: string;
  strategyOverride?: MediaSourceStrategy;
}
```

Backend reconstructs or derives a media request. Frontend applies returned asset URL and saves snapshot.

## 13. Migration Plan

### Wave 0 - Freeze Rules And Docs

- Mark this spec as active target.
- Update `CODE_AGENT_INDEX.md` once implementation begins.
- Keep `IMAGE_FETCH_PERSISTENCE_REFACTOR_PROPOSAL.md` as historical Wave 1 analysis.

### Wave 1 - Contracts And Prompt Contract

- Add `mediaManifest` contracts.
- Update LLM response schema/types.
- Update system prompt to forbid provider URLs.
- Add parser validation for `asset://media`.
- Unit tests for manifest validation.

### Wave 2 - ArtifactMediaOrchestrator

- Create `ResolveArtifactMedia`.
- Move current URL persistence logic into legacy resolver path.
- Implement primary manifest resolver path.
- Replace current direct `ResolveAndPersistHtmlImages` usage in LLM routes with `ResolveArtifactMedia`.
- Ensure snapshots receive already-resolved artifacts.

### Wave 3 - Provider Policy

- Status: implemented backend/API contract baseline on 2026-05-29.
- Added `MediaProviderPolicy` to `PlatformConfig` and `setPlatformConfigSchema`.
- Admin platform config read/update now returns normalized effective policy.
- Stock image resolver now uses the normalized platform provider order.
- Provider status endpoint returns `activeProvider`, fallback mode, provider order, configured key availability, and effective `mediaProviderPolicy`.
- Explicit fallback policy is enforced: if fallback is disabled and the configured provider cannot resolve, Picsum is not applied implicitly.
- Tests added for policy normalization, provider order, and no implicit Picsum fallback.
- Remaining hardening: optional superadmin UI controls and provider connectivity test action can be added on top of the API contract without changing the resolver architecture.

### Wave 4 - Trace And Snapshot Linkage

- Add `MediaResolutionTrace`.
- Write trace records for every media request.
- Add `mediaResolution` metadata to preview snapshots.
- Add validation to prevent active snapshots with unresolved media.

### Wave 5 - Persistent Notifications

- Status: implemented backend/API/UI baseline on 2026-05-29.
- Added shared notification contracts and `SystemNotification` persistence in `system_notifications`.
- Added user API: `GET /v1/notifications`, `PATCH /v1/notifications/:id/read`.
- Added superadmin API: `GET /v1/admin/notifications` with domain/severity/status/project filtering.
- Existing `NotificationPanel` now merges client-local task notifications with unread backend notifications and marks backend notifications read when dismissed.
- Media provider fallback, media resolution failure, media persistence failure, and publish/export blocked paths emit persistent notifications for the user and superadmin audience.
- Remaining hardening: admin UI surface for filtered notifications can be added without changing backend contracts.

### Wave 6 - Edit Mode Unified Regeneration

- Status: backend/API baseline implemented on 2026-05-29.
- Added `POST /v1/projects/:projectId/media/:mediaKey/regenerate` to regenerate from the latest persisted `MediaResolutionTrace` for that key.
- The endpoint reuses the stored semantic query/dimensions from the original manifest request, persists a new regeneration trace, and returns the new internal `/p/media/:assetId` URL.
- Workspace stock regeneration now uses the media-key route when the inspected DOM node carries `data-media-key`, and keeps the legacy stock-regenerate route only as an unkeyed/manual-selection compatibility path.
- Edit-mode regeneration is primary-provider-only: no stock fallback chain and no Picsum fallback. If the configured provider cannot resolve the requested image, the edit action must fail explicitly so the user can choose a different provider/prompt. Provider fallback remains allowed only during full artifact generation/resolution, where the priority is producing a complete first artifact.
- Existing upload/gallery/apply flows are unchanged; the current preview replacement path still saves a new snapshot after applying the returned asset URL.
- Remaining hardening: expose richer trace/media-key metadata in the inspector UI so users can select between historical media keys even when the selected DOM node no longer carries `data-media-key`.

### Wave 7 - Publish/Export Guardrails

- Status: implemented backend guardrail baseline on 2026-05-29.
- Publish and republish scan selected snapshot artifacts before post-processing or file writes.
- Layer 1 export scans selected/active snapshot artifacts before export record creation, screenshot capture, ZIP directory creation, or ZIP writes.
- Unresolved `asset://media/*` placeholders fail explicitly with HTTP-style `409` errors and actionable messages.
- Blocked publish/export emits persistent user + superadmin notifications with unresolved media keys in metadata.
- Tests cover publish, republish, and export blocking before storage writes or record mutation.

### Wave 8 - Tests And E2E

Required tests:

- LLM structured output with `mediaManifest`.
- Placeholder replacement HTML and CSS.
- Provider policy primary selection.
- Fallback notification creation.
- Strict mode failure when provider fails.
- Snapshot metadata contains trace IDs.
- Publish blocks unresolved placeholders.
- Edit-mode regenerate creates asset and new snapshot.
- Existing asset apply/upload still works.

## 14. Regression Guardrails

### Artifact Versioning

Do:

- create snapshots only after media resolution;
- attach trace IDs to snapshot metadata;
- preserve parent snapshot ID;
- preserve selected snapshot behavior;
- never mutate older snapshots in place.

Do not:

- rewrite all historical snapshots automatically;
- resolve media during publication;
- activate failed media-resolution snapshots;
- save unresolved placeholders in active artifacts.

### Publication

Do:

- publish exactly selected/active snapshot;
- scan unresolved placeholders before publishing;
- fail explicitly if unresolved media exists.

Do not:

- call stock/image-gen providers during publish;
- silently fallback during publish;
- publish draft media placeholders.

### Edit Mode

Do:

- reuse existing selected-element flow;
- reuse existing media inspector popup;
- save media changes as new snapshot versions;
- keep upload/gallery/AI image generation controls working.

Do not:

- add a separate standalone media UI;
- bypass existing `saveMediaVersion`;
- mutate DOM without corresponding snapshot save when user confirms change.

### Provider Errors

Do:

- record provider attempts;
- notify fallback;
- notify hard failure;
- expose admin error dashboard.

Do not:

- swallow provider errors;
- treat LoremFlickr fallback as normal success;
- console-only errors for user-impacting media failures.

## 15. Acceptance Criteria

The refactor is complete when:

1. New LLM artifacts use `asset://media/<key>` placeholders.
2. New LLM responses include `mediaManifest`.
3. Backend resolves all manifest media requests before snapshot activation.
4. Every resolved media file is a `ProjectAsset`.
5. Preview snapshots store stable internal media references.
6. Superadmin policy controls stock provider and fallback.
7. Fallback to LoremFlickr creates user and superadmin notifications.
8. Publish/export never resolves media and blocks unresolved placeholders.
9. Edit-mode image controls reuse current popup/accordion flow.
10. Tests cover generation, focus patch, image regeneration, asset apply, snapshot versioning, and publish guardrails.

### 18.3 Known Gaps And Criticalities

For the proposed product/architecture roadmap that addresses these gaps with frontend-selectable strategies, structured JSON planner outputs, and dedicated resolvers, see `docs/specs/MEDIA_STRATEGY_RESOLVER_PIPELINE_SPEC.md`.

- No browser E2E test currently drives the full workspace flow from prompt -> resolved media -> snapshot -> edit regenerate -> publish/export. The backend pieces are covered by targeted tests, but Playwright coverage should be added before declaring product-level completion.
- Failed provider attempts during full artifact generation create notifications/logs, but failed `MediaResolutionTrace` rows are not yet persisted for every exception path. Add failure trace persistence if audit completeness is required.
- Only `auto` and `stock` manifest strategies are implemented. `image_generation`, `project_asset`, and `user_library` still return unsupported-strategy behavior and need dedicated resolvers.
- CSS background media-key continuity depends on the LLM emitting `data-media-key` on the owning HTML element. Foreground/inline HTML placeholders are now auto-annotated during replacement, but stylesheet selector-to-element inference is not implemented.
- The admin notification API exists, but there is no dedicated admin notification dashboard/filter UI yet.
- The edit inspector uses `data-media-key` from the selected DOM node. If later edits remove that attribute, the workspace falls back to unkeyed/manual stock regeneration instead of showing a selectable historical media-key list.
- Media-key regeneration reuses the latest trace by key and optional snapshot; it does not yet expose a user-facing trace picker for multiple historical variants.
- Publish/export guardrails are backend-tested, but browser-level validation should confirm the user-facing error copy and persistent notification panel behavior.
- Provider connectivity testing (`POST /v1/admin/media-provider-policy/test`) remains a spec-level endpoint, not a completed implementation.
