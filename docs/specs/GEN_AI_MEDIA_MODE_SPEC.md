# Gen AI Media Mode Specification

> Status: architectural specification — ready for implementation planning  
> Date: 2026-06-10  
> Scope: extension of the Vibe/Zero Effort/God Mode pipeline to support non-code Gen AI outputs (podcast, video, music) via external engine REST APIs, while preserving the existing HTML artifact container model  
> Audience: maintainers, implementation agents, operators  
> Related specs: `docs/specs/ARTIFACT_MEDIA_ORCHESTRATOR_SPEC.md`, `docs/specs/PRESET_TYPED_SPECS.md`, `docs/specs/EXPORT_AND_PUBLISH_SPEC.md`, `docs/specs/COST_TRANSACTION_LEDGER_SPEC.md`, `docs/specs/ZERO_EFFORT_MEDIA_ASYNC_EVOLUTION_SPEC.md`, `docs/specs/GEN_AI_MEDIA_MODE_INTERFACE_STANDARDS.md`  
> Current media baseline report: `docs/reports/ARTIFACT_MEDIA_IMPLEMENTATION_STATUS_2026-05-29.md`

---

## 1. Executive Decision

The platform shall support **Gen AI media generation** (audio/podcast, video, music) as a first-class extension of the existing Vibe pipeline, not as a parallel product line.

The governing principle is:

```text
The LLM always generates an HTML artifact "container".
The container includes semantic media placeholders (asset://media/<key>).
External Gen AI engines produce the actual media asynchronously.
The backend resolves placeholders into real assets via the existing ResolveArtifactMedia pipeline.
The frontend presents the result in God Mode exactly like any other artifact.
```

This means:

- **No new workspace UI** is required. The existing iframe-based preview, Monaco editor, snapshot history, WYSIWYG edit mode, export ZIP, and publish flows remain unchanged.
- **No new snapshot model** is required. `PreviewSnapshot` continues to store `html`, `css`, `js`.
- **No replacement of Zero Effort** is required. The new capability is delivered as additional presets (`podcast`, `video-show`, `music-track`) inside the existing preset catalog.
- The user experience is **progressive**: the landing page HTML is visible in seconds; the audio/video asset populates the player when the external engine completes.

---

## 2. Non-Negotiable Product Rules

1. Every Gen AI media output must be wrapped in an HTML artifact container. The container is the single unit of versioning, export, and publish.
2. Media assets generated by external engines must be persisted as `ProjectAsset` records, served through `/p/media/:assetId`, and referenced inside artifacts via stable internal URLs.
3. The LLM must never receive raw URLs from external engines. It receives only semantic briefs and generates `asset://media/<key>` placeholders.
4. External engine communication is always backend-to-backend. The frontend never calls the engine directly.
5. All new routes must enforce the existing **double sandbox** (`authMiddleware` + `sandboxMiddleware`).
6. All external engine credentials must be resolved from `process.env` via `apps/api/src/config.ts`. No hardcoded secrets.
7. Generation of long-running media must be **asynchronous**. The user must see a working landing page immediately and receive the media later.
8. Cost tracking must be unified under `CostTransactionService`. Every external engine call generates a `CostTransaction` record.
9. The whitelist in `UploadProjectAsset.ts` must be extended to accept audio/video attachments for user-provided samples/scripts, not removed.
10. Preset definitions remain the single source of truth in `ProjectPreset.ts` (static catalog) with optional MongoDB overlay.

### 3.2 What is missing (gaps)

| Gap | Impact |
|---|---|
| No webhook ingress endpoint | External engines cannot push completion events |
| No generic async job tracking beyond `BackgroundTask` on Conversation | Long-running media jobs need durable status + recovery |
| `UploadProjectAsset` MIME whitelist excludes `audio/*` and `video/*` | User cannot upload voice samples, reference tracks, or video clips |
| `MediaSourceStrategy` lacks `audio_generation` / `video_generation` | `ResolveArtifactMedia` cannot dispatch to an external audio/video engine |
| No external engine REST adapter abstraction | Each engine would be a bespoke `fetch()` block |
| `CostTransaction` has no audio/video EUR rates | Billing cannot track Gen AI media costs |
| Workspace polling only covers image generation and enrichment | Needs a third polling track for audio/video generation status |

### 4.2 Why the HTML Container Is Non-Negotiable

Reusing the artifact container gives the following capabilities **for free**:

| Capability | Cost if we bypass HTML container |
|---|---|
| Snapshot history / versioning | New DB entity + UI panel |
| God Mode inspect + edit | New DOM inspector for audio timeline |
| Export ZIP | New ZIP builder for standalone audio players |
| Publish to public URL | New deployment logic for non-HTML artifacts |
| Focused edit (change title, CTA, bio) | New "patch" semantics for audio metadata |
| WYSIWYG light | New text editor for transcript/show notes |
| Brand identity injection (Layer G) | Re-implementation of logo/color injection |
| Responsive layout | Re-implementation of CSS breakpoints |

The container model treats the podcast **as a website about a podcast**. This is a feature, not a limitation: the user receives a shareable landing page with SEO-friendly text, newsletter signup, host bio, and episode notes — not just an MP3 file.

## 6. Backend Design

### 6.1 New Use-Case: `GeneratePodcastViaApi`

**File:** `apps/api/src/application/use-cases/GeneratePodcastViaApi.ts`

**Responsibility:** accept a structured brief, call the external podcast engine, track the job, and eventually persist the audio file as a `ProjectAsset`.

**Pattern:** Identical to `GenerateProjectImage.ts` (deferred asset + async resolution):

1. Validate sandbox (`userId`, `projectId`).
2. Create a `ProjectAsset` with:
   - `id`: randomUUID
   - `projectId`, `userId`
   - `mimeType`: `"audio/mpeg"` (or inferred from engine response)
   - `label`: brief title
   - `source`: `"platform_generated"`
   - `generationStatus`: `"queued"`
   - `generationMetadata`: `{ engine: "podcast-engine", externalJobId, briefSummary, requestedDuration }`
3. Call the external engine REST API (see Section 9 for contract).
4. If the engine returns a synchronous URL (small engines), download immediately and update asset to `"ready"`.
5. If the engine returns an async job ID:
   - Save `externalJobId` in `generationMetadata`.
   - Update asset to `"processing"`.
   - If the engine supports webhooks: the endpoint is `POST {PUBLIC_API_URL}/v1/webhooks/podcast-engine`.
   - If the engine does not support webhooks: spawn a fire-and-forget polling loop (see Section 8).
6. Return `{ assetId, status, estimatedSeconds }` to the caller.

**Error handling:**
- Engine timeout → asset `generationStatus: "failed"`, emit `SystemNotification`.
- Engine error response → same as above, log to `ExecutionLogger`.
- Partial failure (engine returns audio but no transcript) → asset is `"ready"` but `generationMetadata.transcriptAvailable: false`; transcript can be generated later via a separate LLM call if needed.

### 6.2 New Use-Case: `ResolveAudioGeneration`

**File:** `apps/api/src/application/media/ResolveAudioGeneration.ts`

**Responsibility:** invoked by `ResolveArtifactMedia` when `request.sourceStrategy === "audio_generation"`.

**Interface:**

```typescript
export interface ResolveAudioGenerationInput {
  projectId: string;
  userId: string;
  mediaKey: string;
  request: ArtifactMediaRequest; // includes semanticQuery, kind, role
  conversationId?: string;
  parentSnapshotId?: string;
}

export interface ResolveAudioGenerationOutput {
  assetId: string;
  assetUrl: string; // /p/media/:assetId
  durationSeconds?: number;
  transcript?: TranscriptSegment[];
}
```

**Behavior:**
1. Build a structured brief from the `semanticQuery`, project moodboard, and Layer D context.
2. Call `GeneratePodcastViaApi.execute()`.
3. Return immediately with the deferred `assetId`.
4. `ResolveArtifactMedia` will inject the `/p/media/:assetId` URL into the HTML placeholder.

### 6.3 Extension of `ResolveArtifactMedia`

In `apps/api/src/application/media/ResolveArtifactMedia.ts`:

- Extend the strategy dispatcher to handle `audio_generation` and `video_generation`.
- For these strategies, skip stock-image resolution entirely.
- Emit `media_progress` SSE events with phase `"resolving"` → `"resolved"` even though the asset may still be processing; the URL is stable (`/p/media/:assetId`) and will become playable when the engine finishes.

### 6.4 New Route: Webhook Ingress

**File:** `apps/api/src/presentation/http/routes/webhookRoutes.ts`

**Endpoint:**

```typescript
POST /v1/webhooks/podcast-engine
```

**Body:** `PodcastEngineWebhookPayload` (see Section 9.3).

**Security:**
- Verify `X-Engine-Signature` HMAC-SHA256 using `PODCAST_ENGINE_WEBHOOK_SECRET` (from env).
- Or verify bearer token in `Authorization` header using `PODCAST_ENGINE_API_KEY`.
- Rate-limit by IP (optional, via nginx or express-rate-limit).

**Processing:**
1. Lookup `ProjectAsset` by `externalJobId` (stored in `generationMetadata`).
2. Verify asset exists and `generationStatus !== "ready"`.
3. Download audio from `audioUrl` (presigned temporary URL provided by engine).
4. Stream download directly to `IFileStorage.saveUpload()`.
5. Update asset:
   - `generationStatus: "ready"`
   - `fileSize`, `mimeType`, `storagePath`
   - `generationMetadata.durationSeconds`, `transcript`, `chapters`
6. Emit `SystemNotification` to project owner: "Il tuo podcast è pronto".
7. If the asset is linked to an active `PreviewSnapshot`, optionally trigger a thumbnail refresh (the snapshot page now has playable audio).

**Idempotency:** repeated webhooks for the same `jobId` must be idempotent. Use `generationStatus === "ready"` as a guard.

### 6.5 New Route: Status Polling

**File:** `apps/web/app/workspace/[projectId]/page.tsx`

Add a third polling track alongside the existing image-generation and enrichment polling:

```typescript
// Existing patterns:
// 1. pendingImagePolling (while loop, 2s interval, 45s timeout)
// 2. pendingEnrichmentPolling (setInterval, 3s interval)

// New:
const [pendingAudioGeneration, setPendingAudioGeneration] = useState<string[]>([]);

useEffect(() => {
  if (pendingAudioGeneration.length === 0) return;
  const interval = setInterval(async () => {
    for (const assetId of pendingAudioGeneration) {
      const status = await getAssetGenerationStatus(token, projectId, assetId);
      if (status.generationStatus === "ready") {
        setPendingAudioGeneration(prev => prev.filter(id => id !== assetId));
        incrementPreviewForceKey(); // reload iframe
        toast.success("Il tuo podcast è pronto!");
      }
      if (status.generationStatus === "failed") {
        setPendingAudioGeneration(prev => prev.filter(id => id !== assetId));
        toast.error("Generazione podcast fallita");
      }
    }
  }, 3000);
  return () => clearInterval(interval);
}, [pendingAudioGeneration]);
```

## 8. Async Job Model

### 8.1 Current Limitation

The platform has **no message queue** (no Bull, Redis Queue, or SQS). Async work is done via:

- `setImmediate` fire-and-forget (`SnapshotThumbnailJob`, `AssetEnrichmentPipeline`)
- MongoDB state polling by the frontend (`while` loops on `generationStatus`)

This is sufficient for media generation up to ~10 minutes. For longer jobs, a queue will eventually be needed (see `CODE_AGENT_INDEX.md` M3: BullMQ setup).

### 8.2 Immediate Pattern: Fire-and-Forget Polling

For engines that do **not** support webhooks:

```typescript
// Inside GeneratePodcastViaApi.execute()
setImmediate(async () => {
  const maxAttempts = 120; // 10 minutes at 5-second intervals
  for (let i = 0; i < maxAttempts; i++) {
    await sleep(5000);
    const status = await pollEngineStatus(externalJobId);
    if (status === "completed") {
      await downloadAndPersistAudio();
      await this.assetRepository.updateGenerationStatus(assetId, "ready");
      return;
    }
    if (status === "failed") {
      await this.assetRepository.updateGenerationStatus(assetId, "failed");
      return;
    }
  }
  // Timeout
  await this.assetRepository.updateGenerationStatus(assetId, "failed");
});
```

**Resilience note:** if the API container restarts during polling, the job is orphaned. To mitigate:
- Store `pollAttempts`, `lastPollAt`, and `nextPollAt` in `ProjectAsset.generationMetadata`.
- On API startup, scan for assets with `generationStatus === "processing"` and `nextPollAt < now`, then resume polling.

### 8.3 Future Pattern: BullMQ + GenerationWorker

When `BullMQ` is introduced (M3 in `CODE_AGENT_INDEX.md`), `GeneratePodcastViaApi` should enqueue a `GenerateMediaJob` instead of spawning `setImmediate`:

```typescript
await queue.add("generate-media", {
  type: "podcast",
  assetId,
  externalJobId,
  projectId,
  userId
}, {
  attempts: 3,
  backoff: { type: "exponential", delay: 10000 }
});
```

The spec does not mandate BullMQ for Phase 1, but all code must be written so that the `setImmediate` block can be replaced by a worker without changing the use-case interface.

## 10. Storage & Asset Model

### 10.1 File Storage Paths

| Asset Type | Local Path | MinIO Key |
|---|---|---|
| User upload (voice sample) | `./data/uploads/{userId}/{projectId}/{assetId}.mp3` | `andy-code-cat-media/uploads/{userId}/{projectId}/{assetId}.mp3` |
| Platform-generated audio | `./data/uploads/{userId}/{projectId}/{assetId}.mp3` | same pattern |
| Thumbnail (podcast cover) | `./data/thumbnails/{projectId}/{snapshotId}.jpg` | N/A (thumbnails remain local) |

### 10.2 ProjectAsset Schema Extension

No breaking changes. Only additive fields in `generationMetadata`:

```typescript
interface ProjectAsset {
  // ... existing fields ...
  generationMetadata?: {
    // existing fields already supported:
    conversationId?: string;
    parentSnapshotId?: string;
    mediaKey?: string;
    semanticQuery?: string;
    resolutionRoute?: string;
    fallbackUsed?: boolean;

    // NEW fields for external media generation:
    engine?: "podcast-engine" | "video-engine" | "music-engine";
    externalJobId?: string;
```

## 12. Security & Isolation

### 12.1 Double Sandbox

All new routes must follow the existing pattern:

```typescript
router.post(
  "/v1/projects/:projectId/podcast/generate",
  authMiddleware,
  sandboxMiddleware,
  async (req, res) => { ... }
);
```

### 12.2 Webhook Security

- The webhook endpoint is **public** (no JWT), but authenticated via HMAC or API key.
- It must **not** expose any project data in the response.
- It must validate that the `jobId` belongs to a real `ProjectAsset` before acting.
- Return `200 OK` immediately to avoid retries on success; return `4xx` only for malformed payloads.

### 12.3 URL Validation

When downloading audio from the engine's temporary `audioUrl`, validate:
- Protocol must be `https:`.
- Host must match an allow-list (`PODCAST_ENGINE_ALLOWED_HOSTS`).
- Response `Content-Type` must be `audio/mpeg` or `audio/wav`.
- Response size must not exceed `MAX_AUDIO_DOWNLOAD_BYTES` (env, default 100 MB).

## 14. Risks and Mitigations

| Risk | Level | Mitigation |
|---|---|---|
| **Container restart orphans async polling** | Medium | Store `pollAttempts` / `nextPollAt` in asset metadata; resume on startup scan |
| **Engine unavailable or slow** | Medium | Timeout after 15 min; mark failed; notify user; offer retry button |
| **Webhook spoofing** | Medium | HMAC signature verification + jobId existence check + status guard |
| **Large audio files exhaust storage** | Low | Keep existing quota system; add `MAX_AUDIO_DOWNLOAD_BYTES`; warn in UI |
| **LLM generates broken audio player HTML** | Low | Layer B includes strict HTML template; parser already repairs malformed HTML |
| **Cost explosion on external APIs** | High | Pre-authorize cost at job creation; hard limit per user per day via `PlatformConfig` |
| **Quality of AI-generated podcast** | High | Depends entirely on engine. Mitigation: start with high-quality TTS + script pipelines (e.g., ElevenLabs, Wondercraft, or custom LLM script + TTS two-step) rather than end-to-end podcast generators |
